# 行业案例 PDF 生成器

> **品牌色规范**：主蓝 `#2E56E1`、暖橙 `#E8843A`、暖橙文字 `#A0500E`（不用 `#B65E16`）。中文字体优先使用 ACS/Linux 可用的 `Noto Sans SC` / `Source Han Sans SC`，再 fallback 到微软雅黑、PingFang。

根据客户背景信息，生成一份 3-4 页的精美案例 PDF，苹果风设计，包含 CSS 模拟的系统界面截图。用于微信发送给潜在客户。

## 为什么这样设计

传统的"痛点→方案→价值"三段式方案文档看起来像 AI 批量生成的模板，缺乏说服力。这个 skill 用讲故事的方式呈现案例，通过模拟的系统界面让客户直观看到"用起来长什么样"，比纯文字方案更有冲击力。

核心差异化：**CSS 绘制的模拟系统界面**——看板、数据表格、图表、审批流程等，让客户觉得这是一个真实的产品展示，而不是一篇文章。

## 工作流程

### Step 1: 分析客户信息

从用户提供的信息中提取：
- 公司名称、行业、规模（人数）、所在地
- 核心关注点（最想解决的问题，如投标管理、项目管理、生产管理等）
- 业务形态（制造、销售、工程、贸易、服务等）
- 已有钉钉使用情况（如已用考勤、审批等）
- 特殊需求或补充信息

如果信息不够，向用户确认关键缺失项。但不要过度追问——有公司名+行业+核心关注点就够了。

### Step 2: 确定内容策略

基于客户信息，决定：

**Hero 标题**：场景化表达，不要"XX公司数字化方案"这种官腔。好的例子：
- "从投标混乱到全流程掌控"
- "让每个工地都在掌心里"
- "从手工台账到智能管控"
- "告别跨厂区签字，拥抱移动审批"

**核心场景**（1个，篇幅最大，配主模拟界面）：客户最关注的那个痛点

**补充场景**（1-2个，配辅助模拟界面）：与核心场景相关的高价值场景

**九宫格场景**（9个）：扩展场景，从老板/管理层视角写痛点共鸣

**4个关键数据指标**：优先使用用户提供或已验证的数据；没有验证数据时，必须明确写成“示例目标 / 测算口径 / 预期改善”，不要包装成客户已实现的事实结果。

### Step 3: 生成 HTML

生成一个自包含的 HTML 文件。读取 `references/template-guide.md` 获取完整的 CSS 样式系统和 HTML 组件库。这份参考文件包含所有你需要的 CSS 变量、组件类名、HTML 结构模板和 print 媒体查询，**必须在生成 HTML 前先读取它**。

关键原则：
- 所有模拟界面的数据必须贴合客户行业和地区（福建本地：泉州、晋江、石狮、南安、厦门、福州、漳州、莆田、龙岩等）
- 金额、项目名称、人名要符合行业特征
- 九宫格的文案格式：场景描述 + `<br>` + 加粗的痛点共鸣句（如 `<b>不等结项才发现超支</b>`）
- 不要用"痛点→方案→价值"这种模板结构

HTML 文件存放到 `assets/yyyymmdd/` 目录（8位日期，无分隔符）。文件名用中文，如 `XX行业数字化管理案例.html`。

### Step 4: 转换为 PDF

使用随 skill 提供的 Playwright 脚本将 HTML 转为 PDF：

```bash
python3 SKILL_DIR/scripts/html_to_pdf.py assets/yyyymmdd/案例.html assets/yyyymmdd/案例.pdf
```

`SKILL_DIR` = 本 skill 的绝对路径（即本文件所在目录）。脚本直接用 `file://` 打开本地 HTML 并调用 Playwright `page.pdf()`，不要启动临时 HTTP 服务，不要调用 `localhost:3000/internal/browser/*`，也不要按端口 kill 进程。

生成后用当前环境可用的 PDF/图片检查方式验证页面效果，例如 `pdfinfo` 看页数、`pdftoppm` 导出首页 PNG 后查看。如果分页有问题（内容截断、大片空白、footer 单独成页），调整 `@media print` CSS 后重新生成。

### Step 5: 交付 HTML 预览

先将 HTML 通过 `[FILE]` 标记发送给用户预览（HTML 支持点击直接预览）。简要说明内容结构，等用户确认内容没问题后再进行 PDF 转换。

### Step 6: 用户确认后转 PDF

用户确认 HTML 内容 OK（或提出修改意见并完成修改）后，再执行 Step 4 的 PDF 转换流程。将最终 PDF 通过 `[FILE]` 标记发送给用户。

## 模拟界面组件选择

根据场景选用不同的 CSS 模拟界面（完整代码在 `references/template-guide.md`）：

| 场景类型 | 模拟界面 | 适用于 |
|---------|---------|-------|
| 项目/流程管理 | 看板（Kanban） | 投标管理、项目进度、订单跟踪 |
| 资质/台账管理 | 数据表格 | 证照管理、合同台账、供应商管理 |
| 经营分析 | 数据看板（图表） | 中标率、回款、产值、库存 |
| 流程审批 | 审批流程图 | OA审批、采购审批、报销 |

## PDF 排版要点

PDF 排版是最容易出问题的环节。必须包含完善的 `@media print` CSS：

1. **压缩间距**：网页的大留白在 PDF 上变成浪费空间，print 样式需全面压缩 padding/margin/font-size
2. **分页控制**：
   - `page-break-inside: avoid` 用于模拟界面、表格、图表、九宫格等不可截断的内容块
   - `page-break-before: always` 用于需要独立成页的 section（如九宫格所在的"更多场景"区域）
   - `page-break-after: avoid` + `page-break-before: avoid` 配合使用让 highlight-box 和 footer 不分离
3. **目标 3 页**：第一页（Hero+核心场景+功能卡片）、第二页（补充场景）、第三页（九宫格+highlight+footer）
4. **避免孤页**：footer 不能单独占一页，如果被推到新页，压缩前面的间距

## 九宫格写作指南

这是最容易写得像 AI 的部分，要特别注意。

**错误写法**（功能视角）：
```
审批流程
跨厂区移动审批
领导出差不耽误
```

**正确写法**（老板视角）：
```
审批不卡流程
采购要签字、付款要审批
人在外地手机点一下就过
两个厂区不用跑来跑去
```

区别：正确写法描述的是老板日常遇到的具体场景（"采购要签字"），然后给出共鸣（"两个厂区不用跑来跑去"），而不是在列功能参数。

## 注意事项

- 所有内容用中文
- Footer 固定结构：开沿科技 + 钉钉官方授权服务商 · 泉州本地化服务 + 服务顾问：{当前用户姓名}
  - 服务顾问姓名从当前对话上下文中获取（系统提示语中的用户身份、MEMORY.md 中的用户信息等），不要写死
- 文件命名用中文：如 `制造业投标管理案例.pdf`
- HTML 和 PDF 都输出到 `assets/yyyymmdd/` 目录
- 本 skill 不限于钉钉服务商场景——如果客户的核心需求不是钉钉相关的（如纯定制开发、嗨聘等），应根据实际业务调整 footer 描述和方案内容，不要强行套钉钉
- 行业场景参考表只是起点，要根据客户实际描述的痛点来选择核心场景和模拟界面，不要照搬
