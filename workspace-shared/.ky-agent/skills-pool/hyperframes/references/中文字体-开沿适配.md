# 中文字体 — 开沿适配指南（HyperFrames）

> HyperFrames 内置字体里没有任何中文，所以做中文视频必须自己引入中文字体。首选最省事的「Google Fonts 外链」路径；要产物自包含再走「自带 woff2」路径。写任何中文 composition 前必读本文。

## 1. 为什么要专门处理中文（机制）

1. **内置字体里没有中文**：producer 预置内嵌（data URI、无需 @font-face）的字体里 CJK 只有日文 `noto-sans-jp` / `noto-sans-japanese`，**没有任何中文（Noto Sans SC/TC、思源黑体）或韩文**。
   - 证据：`packages/core/src/lint/rules/fonts.ts`（`PRODUCER_BUNDLED_FONTS` 清单）、`packages/producer/src/services/deterministicFonts.ts`（`FONT_ALIASES` 仅映射到 noto-sans-jp）。
2. **lint 会报 `google_fonts_import`，忽略即可**：这是一条**静态源码规则**——lint 扫描 HTML 文本，只要看到 `fonts.googleapis.com` 这类外链字体写法就报，它**不联网、不验证字体能否访问**，纯粹是规则作者为逼你「字体全部内联 @font-face」而设。它是 warning 级、不阻断渲染，外链字体照样能正常出片，所以对最终视频无影响。（注：这是上游 `@hyperframes/core` 的 lint 行为，本文档改不了它报不报；要它彻底不报得 patch 该包。）
3. **后果**：既不外链、也不自带中文字体 → 中文落回 generic 字体 → 豆腐块（□□□）或字重/字形错乱。

## 2. 路径 A（首选，最省事）：Google Fonts 外链

直接外链 Noto Sans SC 即可，无需下载、无需 `@font-face`：

```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Noto+Sans+SC:wght@400;700&display=swap" rel="stylesheet">
```

CSS 里直接用：`font-family: 'Noto Sans SC', sans-serif;`

- lint 会报 `google_fonts_import`，忽略即可（见第 1 节）。
- 首次用外链做中文视频，`npx hyperframes render` 实渲一次确认中文已正常加载、无豆腐块。

## 3. 路径 B（可选优化）：自带 woff2

当你要产物自包含、渲染更快时，自带本地 woff2：

1. **准备中文 woff2**：思源黑体 / Noto Sans SC（开源可商用）。注意整字库单字重就有数 MB。
2. **放进项目** `fonts/` 目录，与 `index.html` 同级。
3. **写 `@font-face`** 指向本地文件：
   ```css
   @font-face {
     font-family: 'Noto Sans SC';
     src: url('fonts/NotoSansSC-Regular.woff2') format('woff2');
     font-weight: 400;
     font-display: block;
   }
   @font-face {
     font-family: 'Noto Sans SC';
     src: url('fonts/NotoSansSC-Bold.woff2') format('woff2');
     font-weight: 700;
     font-display: block;
   }
   ```
4. **CSS `font-family` 用它**：`font-family: 'Noto Sans SC', sans-serif;`

自带 woff2 不触发 `google_fonts_import` 告警，渲染时也不发外部请求、更快。

### 子集化（路径 B 控体积）

整字库数 MB 会拖慢渲染、撑大产物。文案确定后按实际用到的字做子集：
```bash
# fonttools（pip install fonttools brotli）
pyftsubset NotoSansSC-Regular.otf \
  --text-file=文案.txt \
  --flavor=woff2 \
  --output-file=fonts/NotoSansSC-Regular.subset.woff2
```
- 子集化后体积可从数 MB 降到几十 KB。
- 注意：**参数化/`--variables` 批量场景**，子集必须覆盖所有变体可能出现的字（否则某些卡片缺字）；不确定就用全字库或一个够大的常用字集（如 GB2312 ≈ 6763 字）。

## 4. arm64 渲染注意事项

- **arm64 渲染坑（Issue #294）**：Linux ARM64 + 系统 Chromium 上渲染会报 `HeadlessExperimental.beginFrame`（Chromium 147 移除该协议），需设环境变量 `PRODUCER_FORCE_SCREENSHOT=true` 绕过。开沿 agent 跑在 Mac mini arm64、阿里云也有 arm64 ECS——**首次在这些环境渲染先验这个**。macOS 自带 Puppeteer Chrome 通常不撞，但仍以实测为准。

## 5. 交付前实测清单（中文视频必过）

- [ ] `npx hyperframes inspect` 看每一帧中文**无豆腐块、无缺字**
- [ ] 简体 / 繁体（若有）字形正确
- [ ] 粗体/常规**字重正确**（缺对应字重的 @font-face 或外链 `wght` 会假性加粗或回落）
- [ ] 数字+中文混排对齐正常（必要时 `font-variant-numeric: tabular-nums`）
- [ ] 路径 A（外链）首次实渲一次，确认字体真的加载进来了
- [ ]（`google_fonts_import` 告警可忽略，不算 fail）

## 6. 中文旁白 TTS（豆包，别用 Kokoro 出中文）

视频要中文配音 / 旁白时，**别用 `npx hyperframes tts`**（它底层是本地 Kokoro，中文音素化机械、效果差）。改用本仓自带的豆包脚本（火山引擎 TTS V3，凭证已在 skill 内配好）：

```bash
python3 references/modules/hyperframes-media/scripts/doubao_tts.py \
  "你的中文旁白文案" --voice cancan -o narration.mp3
```

- 输出 24kHz MP3，与 server 端 `integrations/tts/ttsClient.ts` 同一套豆包接口。
- 6 个内置中文音色（cancan/vivi/tianmei/kefu/wennuan/jieshuo），`*_bigtts` 大模型音色支持中英混读，文案夹英文术语发音正常。
- 拿到 `narration.mp3` 后照常 `npx hyperframes transcribe narration.mp3 --language zh` 取词级时间轴。
- 完整音色表与参数见 `references/modules/hyperframes-media/SKILL.md` 的 Text-to-Speech 段。

## 7. 一句话

> 字体：做中文视频**首选 Google Fonts 外链**（省事，lint 告警忽略）；要产物自包含/更快，再走自带 woff2 子集化路径。
> 旁白：中文一律走豆包脚本（第 6 节），不要用 Kokoro 出中文。
